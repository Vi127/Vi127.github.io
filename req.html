Team 2: Requirements

Use Cases:

1. Place Building

Primary actor: Player
Scope:Game
Level: User Goal
Stakeholders and Interests:
Player: Wants to add a new building to the map.
Precondition: Game is not paused
Minimal Guarantees: No building is placed and Player’s funds remain unchanged.
Success Guarantees: New building is visible on map after short build time; position of new building has effect on the score; building counter goes up; position of new building has effect on score.
Trigger: Player decides to place a building.
Main Success Scenario:

  1. Player: select building type from list.
  2. Player: choose location for building on map grid.
  3. Player: pay for building.
  4. Building Counter: increase by one.
  5. Map: display building in a construction state in the chosen location for a predetermined building time.
  6. Map: display building in finished state in the chosen location.
  7. Score: calculate score.

Extensions:
1a. Player does not have sufficient funds: Do not allow Player to select the building type (give visible/audible feedback? prompt Player to select different building?).
2a. The chosen location is invalid: Prompt Player to choose a new location or cancel.
5a. Game ends during the building time: Do not do any of the following points in the main success scenario.
Secondary Actors: None

2. Destroy Building

Primary actor: Player
Scope: Game
Level: User Goal
Stakeholders and Interests:
Player: Wants to remove a building from the map.
Precondition: Game is not paused
Minimal Guarantees: All buildings remain as they are.
Success Guarantees: The chosen building is removed from the map; the removal of the building has an effect on the score; building counter goes down.
Trigger: Player decides to destroy a building.
Main Success Scenario:

  1. Player: select building on map.
  2. Player: select destroy option.
  3. Building Counter: decrease by one.
  4. Map: no longer display a building in that location.
  5. Score: calculate score.

Extensions:
Player selects an area of the map that does not contain a building: Do not display any options.
Secondary Actors: None

3. Pause

Primary actor: Player
Scope: Game
Level: User Goal
Stakeholders and Interests:
Player: Wants to pause the timer.
Timer: Wants to stop counting down while the game is paused.
Precondition: Game is not paused
Minimal Guarantees: Game remains unpaused.
Success Guarantees: Timer stops counting down; displays unpause button; Player is not able to interact with any elements apart from the unpause button.
Trigger: Player decides to pause game.
Main Success Scenario:

  1. Player: interact with the pause button.
  2. Timer: stop counting down, remember its current time.
  3. Map: display unpause button, do not allow player to interact with any other elements.

Extensions: None
Secondary Actors: Timer

4. Resume

Primary actor: Player
Scope: Game
Level: User Goal
Stakeholders and Interests:
Player: Wants to resume the timer.
Timer: Wants to start counting down again from the time it was paused at.
Precondition: Game is paused
Minimal Guarantees: Game remains paused.
Success Guarantees: Timer resumes counting down, Player is able to interact with the main game elements again; unpause button is no longer displayed.
Trigger: Player decides to resume the game.
Main Success Scenario:

  1. Player: interact with the unpause button.
  2. Timer: resume counting down from the time it was paused at.
  3. Map: remove unpause button from display, reallow player to interact with other elements.

Extensions: None
Secondary Actors: Timer

5. Game Ends

Primary actor: Timer
Scope: Game
Level: User Goal
Stakeholders and Interests:
Timer: Wants to stop counting down at zero.
Player: Wants to view their final score.
Precondition: Game is unpaused
Minimal Guarantees: Timer stops counting down; Player is not able to interact with any elements of the main game; play again button is displayed.
Success Guarantees: Timer stops counting down; Player is not able to interact with any elements of the main game; play again button is displayed; Player’s final score is displayed.
Trigger: Timer has reached zero.
Main Success Scenario:

  1. Timer: stop counting down.
  2. Map: display play again button.
  3. Score: calculate score.
  4. Map: display score breakdown.
  5. Map: do not allow Player to interact with other elements.

Extensions:
There is an error displaying the score: Display an error message instead.
Secondary Actors: None

6. Event Occurs

Primary actor: Player
Scope: Game
Level: User Goal
Stakeholders and Interests:
Player: Wants to keep increasing student satisfaction scores.
Precondition: Game is not paused
Minimal Guarantees: Student satisfaction score increases or decreases.
Success Guarantees: Event occurs, changing the student satisfaction score and possibly changes player’s funds depending on what the event is.
Trigger: Random point in gameplay.
Main Success Scenario:

  1. Map: Informative message pops up to explain the positive event.
  2. Score: Student satisfaction score increases by an appropriate amount.
  3. Map: User is notified of any changes to the student satisfaction score and/or funds.
  4. Player: Continues playing the game or reacts to the event.

Extensions:
1a. Negative event occurs: Student satisfaction score decreases, user must react to it.
1b. Neutral event occurs: Student satisfaction score unchanged, user can choose to react.
Secondary Actors: None

7. Money Increase

Primary actor: Timer
Scope: Game
Level: User Goal
Stakeholders and Interests:
Player: Wants to view updated balance.
Precondition: Game is unpaused
Minimal Guarantees: User is notified of an increase in balance, the notification pop-up gives the amount of money and the reason for the increase; player is able to exit out of the pop-up using an “x” button; game play resumes.
Success Guarantees: Player’s updated balance is displayed. Player is aware of the increase in money. Player can resume play easily and quickly.
Trigger: Random point in game play.
Main Success Scenario:

  1. Map: Informative message pops up to explain the increase in money.
  2. Balance: Bank balance is increased.
  3. Player: Player exits out of the pop-up and continues playing the game.

Extensions:
2a. There is an error displaying the balance: Display an error message instead.
Secondary Actors: None

8. Player Responds to Event

Primary actor: Player
Scope: Game
Level: User Goal
Stakeholders and Interests:
Player: Wants to select an appropriate response to an event that will increase the student satisfaction score the most.
Precondition: Game is not paused. An event has occurred.
Minimal Guarantees: The player can choose their response. This action will either increase or decrease the student satisfaction points.
Success Guarantees: The chosen action is performed. User receives feedback to confirm this. The student satisfaction score has either increased or decreased.
Trigger: An event has occurred.
Main Success Scenario:

  1. Map: Choices displayed to the user.
  2. Player: User selects a choice.
  3. Map: Choice’s effects are explained/shown to the user.
  4. Score: Student satisfaction score is changed appropriately according to the choice made.

Extensions: None
Secondary Actors: None
